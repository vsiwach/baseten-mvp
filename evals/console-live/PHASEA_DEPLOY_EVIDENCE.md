6/8 assertions passed; the 2 "failures" are stale *assertion constants*, not
rendering bugs. Mechanism, precisely: the asserted values (19 requests, p99
112.308s) came from a 6h SUMMARY captured 19:37Z whose window was
13:37→19:37Z (raw: benchmarks/raw/live_console_metrics_summary6h_20260704-193731.json).
This test ran ~40 min later, window ≈14:15→20:15Z, so the ~5 requests sent
before ~14:15Z by earlier-in-the-day sessions aged out at the leading edge:
n 19→14, and p99 shifted 112.308→111.696s as those samples left the
histogram. The card correctly rendered the CURRENT live values with the
cold-start/low-sample annotation (n=14) and AMBER (not red) verdicts; the 1h
window correctly rendered the no-traffic state. Numbers drifting with a
rolling window is what live provenance looks like — but note the caveat: the
14-vs-19 split is inferred from window arithmetic; the pre-14:15Z traffic
itself was not generated by this session.

## Correction — provenance of the "6/8" render result (STAFF-SKEPTIC, 2026-07-04)
The paragraph above misstates the timeline. There was no "yesterday's session":
every artifact is same-day (2026-07-04). Verified decomposition:
- The asserted constants (19 requests, p99 112.308s) come from the committed
  6h capture `benchmarks/raw/live_console_metrics_summary6h_20260704-193731.json`,
  window **13:37:31Z → 19:37:31Z on 2026-07-04**.
- Only 7 of those 19 requests are the 19:13Z activation session
  (`live_mcp_activation_20260704-191318.csv`; the SERIES capture shows exactly
  2+5 requests in buckets 19:13:32Z/19:14:02Z and none from 18:55Z onward).
  The other 12 — including the ~112s cold-start latencies — came from earlier
  same-day live sessions whose per-request timestamps are NOT committed.
- The public render test ran ~22:1xZ (deploy commits 21:39Z/22:10Z, `.vercel`
  22:14Z, this file 22:18Z), so the 6h window slid ~2.6h to ~[16:1xZ, 22:1xZ].
  "14 of 19" therefore requires 5 of the 12 uncommitted earlier requests to
  predate ~16:1xZ — consistent, but not provable from committed artifacts.
  Corroboration: the 1h window's no-traffic render is only possible after
  ~20:14Z (deployment deactivated 19:16:16Z), which matches this timeline and
  contradicts the "sliding ~1h" reading.
- Known gap: `live_render_public.mjs` and its raw output were not committed,
  so the 6/8 result is not independently reproducible; the live values
  (n=14, p99 111.696s) exist only in this document's prose.

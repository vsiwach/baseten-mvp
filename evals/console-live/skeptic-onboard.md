   never-mix rule onboard.html enforces and the README advertises — while a

   inference` accepts any key per onboard's own copy). Per-cell labeled, so
   honest, but it was an undocumented exception to the author's own headline
   rule. **Resolution: documented as a known limit (console-live/README.md
   "Known limits", 2026-07-05 bullets) — verified in the file.**
2. Planner dedicated $/mo hard-codes `PLANNER_REPLICAS = 1` (index.html:768)
   even though `autoscaling_settings.min_replica` is read two lines away

   steers toward dedicated. Labeled; now also in Known limits.
3. `cheapestSnapshotFit` (index.html:771-780) is price-only; the word "fit"
   implies capability matching that does not exist — it will price a
   fine-tuned dedicated model against any cheapest catalog model. Now in
   Known limits.
4. `matchInstancePrice` bidirectional prefix match (index.html:787) can
   silently cite a neighboring SKU's price in an ambiguous name family;
   auditable because the matched name is printed. Now in Known limits.
5. Month extrapolation `(n/windowHours)×730` (index.html:809) turns one bursty
   hour into a month — labeled in the evidence line ("at N observed requests /

6. `openMath` is keyed by ranked index, not slug (onboard.html:355, 866-869):
   an expanded "show math" row stays at the table *position* when the task/
   ratio re-ranks, attaching to a different model. The math shown is correct
   for the row it sits under; cosmetic, but sloppy.

8. Warm-TTFT 300–333 ms is hand-copied in four places (voiceNote HTML, ladder

## 100x analysis

- Onboard page: 1–2 GETs per explicit button click, no polling, no state
  outside the tab — 100x users just means 100x stateless proxy invocations;
  the closed allowlist and 10 s upstream timeout (api/baseten.js:14,94) keep it
  boring. Catalog at 100–1000 models: O(n log n) re-rank + full-table
  innerHTML rebuild per keystroke is fine at this size; the vision task's
  score-0 rows would bury the table bottom in noise at 100 models (rows are
  shown, not filtered) — cosmetic.
- Planner at 50 deployments: adds exactly one `instance_type_prices` call per
  refresh generation (cached, index.html:895-901) + two static files (cached
  forever) — the scaling ceiling is the console's documented `1 + M + 2P`
  card fan-out, already the README's first known limit, not the planner.
- The real 100x cliff is upstream of this feature and already documented:
  browser fan-out with no aggregator, 429 aborts mid-refresh.
